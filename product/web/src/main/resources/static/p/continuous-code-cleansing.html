Continuous Code Cleansing

A bit of history

Once upon a time, we developed software in a waterfall process.

We wrote requirements specifications, functional specifications, design specifications, test specifications.

Then we started coding. We applied individual code ownership, and ended the development phase by trying to integrate all the parts that had been developed in isolation by the different developers.

This tended to be a nightmare to manage. No-one could tell how long the integration phase would take, and what problems might pop up.

So the developer community invented Continuous Integration, CI. Now we integrate early and often, problem solved.

With CI in place, the next thing to become a pain was delivery.

Each time we tried to deploy our software to production was like a lottery. It was not uncommon that deployment of a new version could take days. Many times it failed, and we had to roll back to the previous version.

So the developer community invented Continuous Delivery, CD. Once again, problem solved.

And now what?

As lean and agile developers we are now quite good at delivering working software with sustainable velocity.

Sprint after sprint we deliver new shiny features into production. It is not very common though that we kill features!

We must soon start thinking about killing features. We must start doing Continuous Code Cleansing, CCC, the next logical step after CI and CD.

Unless we continuously clean up after ourselves, our code base will inevitably grow into a Big Ball Of Mud, no matter our good intentions and high skill level.
